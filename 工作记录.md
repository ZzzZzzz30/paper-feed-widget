# PaperFeed 工作记录

## 2026-05-24 14:30 — 翻译安全加固（P0修复）

### 问题1 (P0): `allow_translation_fallback` 默认值与设置页显示不一致
- **文件**: `electron/services/translation/service.ts`
- **原因**: `service.ts` 中 `getSetting('allow_translation_fallback', 'true')` 默认为 `'true'`，但设置页 UI 显示"禁止"，实际翻译逻辑却允许降级，导致腾讯云失败后可能自动调用阿里云产生费用
- **修复**: 改为 `getSetting('allow_translation_fallback', 'false') === 'true'`，与设置页一致

### 问题2 (P0): Ollama 被无条件加入翻译链，"禁止降级"语义不干净
- **文件**: `electron/services/translation/service.ts`
- **原因**: 不论用户选择哪个引擎、是否允许降级，Ollama 始终被 `chain.push` 加入，导致"禁止降级"时仍会尝试 Ollama
- **修复**: Ollama 现在仅在以下情况加入链：①用户选择 Ollama 作为主引擎；②主引擎是腾讯云/阿里云且 `allowFallback=true`
- **新逻辑**:
  - 腾讯云 + 禁止降级 = 只用腾讯云
  - 阿里云 + 禁止降级 = 只用阿里云
  - Ollama = 只用 Ollama
  - 腾讯云 + 允许降级 = 腾讯云 → 阿里云 → Ollama

### 问题3 (P0): 用量记录在调用云端 API 之后，存在「调用成功但未记录」和「并发穿透」风险
- **文件**: `electron/services/translation/service.ts`
- **原因**: 旧流程为 `checkLimit() → await API → recordTranslation()`，如果标题翻译成功但摘要失败，整体 throw 后 `recordTranslation` 不执行 → 本地统计低估；如果并发翻译，两篇同时通过 `checkLimit` 后可能双双突破限额
- **修复**: 改为「调用前预占额度」— 新增 `reserveTranslationUsage()`，在执行云端 API 前先完成限额检查和用量记录写入。即使远程调用后续失败，本地统计偏高也比偏低安全

### 问题4 (P0): 腾讯云欠费状态仅用于设置页显示，未在翻译前阻止
- **文件**: `electron/services/translation/service.ts`
- **原因**: `checkTencentOpenStatus()` 只在 `translation:quota-overview` IPC 中调用用于设置页显示，实际翻译 `translateArticle()` 不检查欠费状态
- **修复**: 新增 `ensureTencentAvailable()` 函数，带 5 分钟 TTL 缓存，在实际调用腾讯云翻译前检查 `hasOpen` 和 `hasArrearage`。欠费/未开通/查询失败时 `fail-closed` 并暂停该 provider

### 问题5 (P1): `initial_used_chars` 未按月份隔离，上月设置的值延续到本月
- **文件**: `electron/services/translation/service.ts`, `src/components/SettingsPanel.tsx`, `electron/ipc-handlers.ts`
- **原因**: `initial_used_chars` 是全局设置，用户 5 月设为 100 万表示用完，6 月仍显示 100 万
- **修复**:
  - 新增 `_initial_used_month` 设置项，与月份绑定
  - `getMonthlyInitialWan()` 仅在 `savedMonth === currentMonth` 时返回 `initial_used_chars`，否则返回 0
  - 设置页修改 `initial_used_chars` 时同步写入 `_initial_used_month`
  - 仪表盘 IPC 同样按月份读取初始值

### 问题6: 设置页删除 DeepSeek 价格/预算/用量卡片
- **文件**: `src/components/SettingsPanel.tsx`
- **修复**: 删除 DeepSeek 输入价/输出价/本月预算/初始已用输入框 + `<DeepSeekUsage />` 卡片组件

### 问题7: 云端状态实际查询
- **文件**: `electron/ipc-handlers.ts`
- **原因**: 仪表盘的腾讯云云端状态是硬编码字符串
- **修复**: 实际调用 `checkTencentOpenStatus()`，显示真实的开通/欠费状态

### 问题8: AI对话框同篇文章恢复历史会话
- **文件**: `src/components/AnalysisDialog.tsx`
- **修复**: 恢复 `getAnalysisSession(articleId)` 调用，退出再打开保留对话历史

## 2026-06-16 21:42 +08:00 — 工作记录复核与补充修复

### 复核结论
- **问题1/2/3/4/6/7**: 已在当前代码中看到对应实现，主体修复成立
- **问题5**: 月份隔离逻辑已实现，但存在入口不一致风险；设置页“额度用完”按钮只写入 `*_initial_used_chars`，没有同步写入 `*_initial_used_month`，在按月隔离后可能导致“标记额度用完”不生效或刷新后显示不一致
- **问题8**: 前端已恢复 `getAnalysisSession(articleId)` 调用，但 `sql.js` 数据库 wrapper 的 `run()` 固定返回 `lastInsertRowid: 0`，导致新建 AI 会话时 `sessionId` 可能为 0，历史会话保存和恢复不可靠

### 补充修复1: 修正 sql.js `run()` 返回值
- **修改时间**: 2026-06-16 21:42 +08:00
- **文件**: `electron/services/db.ts`
- **修复**: `StatementWrapper` 持有底层 `SqlJsDb`，`run()` 执行后通过 `getRowsModified()` 返回变更行数，并用 `SELECT last_insert_rowid()` 返回真实插入 ID
- **影响**: `analysis_sessions` 新建后可以得到真实 `sessionId`，AI 对话历史恢复链路才可靠

### 补充修复2: 初始已用月份绑定下沉到主进程
- **修改时间**: 2026-06-16 21:42 +08:00
- **文件**: `electron/ipc-handlers.ts`
- **修复**: `settings:update` 检测到 `tencent_initial_used_chars` 或 `aliyun_initial_used_chars` 更新时，自动同步写入对应 `*_initial_used_month = 当前月份`
- **影响**: 不管设置页哪个入口或未来其他入口修改初始已用量，都会自动绑定当前月份，避免 UI 与实际限额判断分叉

### 补充修复3: 重置统计语义完整化
- **修改时间**: 2026-06-16 21:42 +08:00
- **文件**: `electron/ipc-handlers.ts`
- **修复**: `translation:reset-stats` 除删除本月 `translation_usage` 外，同时清零对应 `*_initial_used_chars`、写入当前 `*_initial_used_month`，并清除该 provider 的暂停/警告控制状态
- **影响**: 用户点击“重置统计”后仪表盘与实际翻译控制状态一致，不会因为初始已用量残留继续显示接近限额或暂停

### 补充修复4: “额度用完”按钮同步月份
- **修改时间**: 2026-06-16 21:42 +08:00
- **文件**: `src/components/SettingsPanel.tsx`

- **影响**: 按钮点击后立即参与当月限额计算，仪表盘刷新结果与用户操作一致
